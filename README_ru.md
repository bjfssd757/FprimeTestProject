# Введение

F` - это фреймворк от NASA JPL, используемый в эмбедед разработке, в том числе в рамках миссии Mars 2020 как фреймворк, на котором работал бортовой компьютер вертолёта Ingenuity.

Основная работа фреймворка построена на основе компонентов, их взаимодействия между собой, а также с GDS (Ground Data System) - своего рода наземная станция, реализованная в F` как Python приложение.

<details>
    <summary>Немного про архитектуру F`</summary>

    - Связь между компонентами обеспечивают порты
    - Передачу текущих ключевых значений от компонентов на GDS обеспечивает телеметрия
    - Передачу значений от GDS на компоненты обеспечивают параметры
    - Передачу команд от GDS на компоненты обеспечивают, соответственно, команды
    - Передачу оповещений от компонентов на GDS обеспечивают события

Одно из главных преимуществ F` - это декларативный язык FPP (F Prime Prime), в котором описывается сигнатура компонента (компонентов), а затем он генерируется в C++, позволяя написать саму логику компонента.

---

GC (Guidance and Control) - это система наведения и контроля, позволяющая спутникам чётко следовать заданной ЦУПом ориентации.

Обычно реализуется целиком система G**N**C (G., Navigation and C.), но навигация требует специфических знаний о КА, поэтому для сохранения универсальности я реализовал только GC часть.

# Архитектура

В первую очередь все данные поступают от датчиков в преобразованном виде от компонента навигации - это основа. Данная реализация GC, принимая данные от навигационного компонента, использует фильтр EMA (экспоненциальное среднее) для сглаживания шумов, но для большей уверенности рекомендую предварительно прогнать данные через нелинейный фильтр, например, расширенный фильтр Калмана (EKF).
Все полученные векторы компонент сразу же нормирует для дальнейшего использования в PD регуляторах. Эта операция требует взятия квадратного корня при нормализации, но она происходит только один раз при поступлении данных.

## PD-регулятор

*Почему нет P**I**D-регулятор?* - Дело в том, что интегральная часть регулятора может накапливать ошибку, что критично для системы с риском насыщения исполнительных органов (в нашем случае - маховики).

В компоненте используется 3 версии регуляторов:
- Для просчёта момента катушек
- Для просчёта комбинированного момента (маховики + катушки)
- Для просчёта момента маховиков

Зачем? Тут ответ будет *насыщенным*...
Маховики позволяют регулировать угол с высокой точностью, но у них есть очень неприятный эффект: *насыщение* (или же сатурация). При достижении маховиком предельной скорости вращения, он уже не может повлиять на поворот спутника - это и есть насыщение.
Для избежания этого эффекта (т.е. для десатурации маховиков) используют дополнительную систему ориентации. Т.к. данная реализация GC рассчитана на низкую околоземную орбиту, тут дополнительной системой выступают катушки.
С Земли (точнее GDS) задаётся параметр максимального и минимального процента насыщения маховиков. Если сатурация маховика ниже минимального процента, за ориентацию отвечают только они. Если же она между минимальным и максимальным процентом, то для поворота вычисляется момент и для маховиков, и для катушек (чем выше сатурация, тем больше момента отдаётся катушкам для разгрузки маховиков). Если сатурация выше максимального процента, маховики вообще перестают участвовать в системе ориентации, система переходит в [режим стабилизации угловых скоростей](#режим-удержания-угловых-скоростей-detumble), а за регулирование поворотов отвечают только катушки.


## Режимы работы

В данной реализации GC предусмотрено несколько режимов работы системы.
Переход между состояниями осуществляется в первую очередь самой системой, но его также можно регулировать с GDS через команды.

### Режим следования точки (Pointing)

Данный режим применятся 90% времени всей миссии. GDS задаёт целевую точку, а GC рассчитывает моменты для маховиков и (при необходимости их разгрузки) для катушек, чтобы к определённому времени иметь заданную угловую скорость и ориентацию.

### Режим удержания угловых скоростей (Detumble)

Данный режим, при нормальном функционировании системы, включается только с GDS командой.

Основная цель - свести угловые скорости к минимуму и удерживать их, пока активен режим.
Особенности:
- игнорирование текущей целевой точки
- применение только катушек

### Безопасный режим (Safe)

Данный режим включается при серьёзных сбоях в памяти, когда восстановить повреждение уже нельзя (см. [ECC](#код-коррекции-ошибки)); при низком уровне заряда батареи; при ручном переключении состояния с GDS.

Основные цели безопасного режима - не войти в резонанс, экономия энергии, дать батарее зарядиться.
Особенности:
- игнорирование текущей целевой точки
- поворот спутника батареями к Солнцу (т.е. сведение нормали панелей к нормали на Солнце)
- использование для ориентации только катушек, т.к. маховики сильно расходуют батарею.

### Выключенный режим

В данном режиме никакие расчёты не производятся. Входные порты игнорируются, в выходные ничего не записывается


## Код коррекции ошибки

Также известный как ECC (Error Correction Code) - чаще всего аппаратная реализация обнаружения и корректировки ошибок памяти, вызываемых попаданиями заряженных частиц в чип.
В данном компоненте предусмотрена программная реализация класса-контейнера для ECC на основе голосования большинства.
Класс хранит в себе 3 копии и 1 зеркальную (битовая инверсия) версию одного и того же значения (TMR). И копии, и зеркало выравниваются по 64 бита (`alignas(64)`) из-за чего каждая версия лежит на разных линиях кэша процессора, что в теории должно спасти, если частица попадёт в границу между линиями, повредив только 1 копию, которая потом легко восстанавливается.
Данный контейнер даёт значительный оверхед и по памяти (в 4 раза больше, чем по факту занимает объект), и по скорости чтения (**важная оговорка**: значительный оверхед по чтению происходит только тогда, когда основная копия не совпала с инвертированным значением зеркала, т.е. либо копия, либо зеркало заведомо повреждено. В таком случае класс контейнер пытается восстановить значения), и даже по скорости записи (не одна операция записи, а 4 + ещё надо вычислить зеркало, хотя это довольно быстро), поэтому защищать этим контейнером имеет смысл только те переменные, которые должны храниться достаточно долго (если перезапись идёт каждые 10мс, но защищать просто не имеет смысла, т.к. ошибка просто перезапишется, если она вдруг произойдёт, но вот если чтение происходит чаще записи, то защита уже имеет место быть).
Основная проблема возникает, если повреждено слишком много копий. В таком случае восстановить данные мы уже не можем и их приходится перезаписывать переданным значением по умолчанию (передаётся при каждой get операции). GC в таком случае не падает, но, если повреждён ключевой объект системы, компонент переходит в безопасный режим.
